8K cloud-workflow validation

Prism

Prism is a precision instrument for validating end-to-end 8K video workflows in the cloud. It originates an instrumented stream, encodes and transmits it, and measures the true glass-to-glass latency at every stage of the delivery chain — proving that a full-resolution 8K pipeline on cloud GPUs holds real time and stays inside its latency budget. Because Prism stamps time-zero into every frame, it can measure all of this without guessing.

Freshness

Precise stages catch the exact frame (frame-level age); freshness stages read the newest served frame. A warm-up window keeps pipeline spin-up out of the numbers, and every run resets to zero, so results reflect steady state — not the first noisy seconds.

Capabilities

What it does

A complete test rig in one toolset — origination through playback, with the instrumentation an ordinary encoder can't provide.

Originate

Generate the source

A test pattern, a pre-rendered file, or the instrumented QR-timecode source that carries the measurement. Any resolution up to 8K.

Encode

Hardware or software

NVENC H.264 / HEVC / AV1 or libx264/libx265, with presets, rate control, and split-frame encoding across multiple NVENC engines for 8K. Never falls back to CPU silently.

Deliver

CMAF, SRT, managed origin

CMAF (HLS + DASH) to disk, an SRT contribution, and ingest into MediaPackage v2 — which serves HLS, LL-HLS, and DASH from one chunk set, in a two-variant ladder (8K + 1080p preview).

Measure

Per-frame latency

QR-timecode latency across T0→T5 — precise where the exact frame can be caught, freshness where only the newest segment is visible.

Span machines

Distributed probes

Passive probes on separate instances measure a real cross-network SRT contribution — not a loopback — with clock-synced one-way timing.

Operate & observe

Console + Player

An always-on console (auth, fleet control, remote campaigns, live T0→T5 graph) and a hosted player with deep playback telemetry. Every run writes a diffable JSON + CSV report.

What you put it to

What it lets you test

The questions a delivery pipeline actually raises — answered on real hardware and a real network, with numbers you can trust and compare.

Throughput

Real-time headroom

Can this GPU hold 1× (or N×) at a given resolution, frame rate, codec, and preset? Live speed, fps, and drop telemetry answer it on the actual instance — e.g. 8K60 HEVC at 1.0×.

End to end

Glass-to-glass latency

The true source→viewer delay, with a per-stage breakdown — encode vs. SRT vs. packaging vs. delivery vs. player — so you can see which latency dominates.

Across the wire

Cross-machine latency

Measure a genuine SRT contribution between instances, over the real network, with the one-way timing that only synchronized clocks make honest.

Trade-offs

Codec & config A/B

Sweep any mix of codec, preset, resolution, frame rate, segment duration, and bitrate; every report embeds its configuration and diffs directly against another.

Integrity

Timecode survivability

Confirm the QR survives the encode at a given bitrate and resolution before trusting any number — a pass/fail gate, not a guess.

Integrity

Known-delay calibration

Inject a measured delay at one stage and confirm that only that stage moves — the correlation check that makes the whole chain believable.

The rig

Hardware & cloud

The validated results run on AWS EC2 — a small fleet in one VPC — but nothing here is AWS-specific.

Encoder

NVIDIA L40S · g6e

Three NVENC engines drive the split-frame 8K60 encode; runs the headless campaign engine the console relays to.

Mux + packager

NVIDIA L4 · g6

Receives the SRT contribution, packages CMAF / HLS, pushes the two-variant ladder into MediaPackage, and transcodes the 1080p preview — without ever touching the encoder's pace.

Control plane

Console · t3.small

Always on: TLS + per-user auth, the pipeline graph, fleet start/stop, campaign relay, and the hosted player. Survives fleet power-cycling.

AWS glue

VPC · Time Sync · MediaPackage

Private-IP fleet in one VPC; the Amazon Time Sync Service keeps chrony µs-accurate (what makes cross-machine timing honest); MediaPackage v2 is the managed origin tier.

Not AWS-locked

Prism is Python + FFmpeg + NVENC/NVDEC, so it runs on any Linux GPU host. The AWS pieces map straight across: EC2 GPU instances → GCP A3/G2 or Azure NV/NC series; the Time Sync Service → any chrony/PTP/NTP source; security groups → the platform's firewall controls; MediaPackage → any LL-capable origin. Same rig, different cloud.

Where it stands

Validated so far — and what's left

The measurement chain is proven end to end at 8K, through a managed origin, to a real player. What remains extends geographic coverage and resilience — none of it blocks the numbers that already work.

Validated

7 confirmed

Demonstrated on real hardware, with pass/fail gates where it counts.

  • Timecode survivability 600 / 600 frames (100%) decoded through 8K HEVC NVENC — the QR carrier survives the encode.
  • Correlation integrity a known +500 ms delay injected at the encode and SRT stages rose alone (+471 ms and +523 ms respectively), with no other stage affected. The injection gate is fully closed.
  • Encode holds real time 8K60 HEVC NVENC, split-frame across three engines — sustained 1.0× with zero dropped frames.
  • Distributed chain, correctly ordered T0→T3 across two machines nests as it should — ~3.0 s encode, +0.4 s SRT, +0.1 s packaging, ~3.5 s glass-to-glass.
  • Managed origin ingest into MediaPackage v2 without MediaLive; the QR survives re-packaging (36/36 decoded by the unmodified probe); LL-HLS with real partial segments and DASH emitted from one chunk set.
  • Player closes the chain the Prism Player decodes the QR at playback and reports T5 — every stage T0→T5 measured in one live run, including the delivery tier (T4) at the MediaPackage egress.
  • Cross-machine clocks µs-synced to the AWS time source, so one-way SRT timing is trustworthy.

Still to validate

2 open

Coverage and resilience — neither blocks the core numbers.

  • Per-region CDN (T4 at the edge) today's T4 reads the MediaPackage egress; the next step puts CloudFront in front and measures freshness from probes in multiple regions — real geographic propagation.
  • Encoder redundancy & sync run a hot-standby encoder frame- and timecode-aligned with the primary, and exercise the failover workflow end to end.

Broadening the instrument

Planned

More formats and modes for the pipelines you actually run.

  • Chunk-aware probing — sub-segment latency visibility so low-latency delivery can be measured at part granularity, not just per segment.
  • DASH-MPD live measurement — probe DASH manifests directly; endpoint polling is HLS-only today.
  • Audio + A/V sync — carry an audio track and measure audio latency and lip-sync alongside video.
  • VVC (H.266) support — bring the next-generation codec into the encoder lineup.

Honest by design

Every stage is labeled measured, derived, or not measured — Prism never presents an estimate as a measurement. T1/T2 are measured per frame on the instrumented path (T2 is sample-sparse at 8K by design); T3–T5 are freshness readings taken wherever a real endpoint serves. What it can't observe cleanly, it says so.
