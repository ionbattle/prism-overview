8K cloud-workflow validation

Prism

Prism is a precision instrument for validating end-to-end 8K video workflows in the cloud. It originates an instrumented stream, encodes and transmits it, and measures the true glass-to-glass latency at every stage of the delivery chain — proving a full-resolution 8K pipeline on cloud GPUs holds real-time and stays inside its latency budget. Because it stamps time-zero into each frame, it's the one component that can measure that without guessing.

The measurement model

One timecode, followed across six stages

Every frame carries a QR timecode stamped with its emission epoch. Latency at any point is that frame's own age when it's seen there — no cross-clock arithmetic, no assumptions.

Capabilities

What it does

A complete test-rig source in one tool — origination through delivery, with the instrumentation a normal encoder can't provide.

Originate

Generate the source

A test pattern, a pre-rendered file, or the instrumented QR-timecode source that carries the measurement. Any resolution up to 8K.

Encode

Hardware or software

NVENC H.264 / HEVC / AV1 or libx264/libx265; presets, rate control, and split-frame encoding across multiple NVENC engines for 8K. Never silently falls back to CPU.

Deliver

CMAF and/or SRT

CMAF (HLS + DASH) to disk, an SRT contribution, or both from a single encode via the tee muxer — one decode, two sinks.

Measure

Per-frame latency

QR-timecode latency across T0→T5 — precise where the exact frame can be caught, freshness where only the newest segment is visible.

Span machines

Distributed probes

Passive probes on separate boxes measure a real cross-network SRT contribution — not a loopback — with clock-synced one-way timing.

Observe & report

Live view + records

Live telemetry, a T0→T5 pipeline graph, campaign results, source preview, and an in-browser player. Every run writes a diffable JSON + CSV.

What you put it to

What it lets you test

The questions a delivery pipeline actually raises — answered on the real hardware and network, with numbers you can trust and compare.

Throughput

Real-time headroom

Can this GPU hold 1× (or N×) at a given resolution, fps, codec, and preset? Live speed, fps, and drop telemetry answer it on the actual instance — e.g. 8K60 HEVC at 1.0×.

End to end

Glass-to-glass latency

The true source→viewer delay, with a per-stage breakdown — encode vs SRT vs packaging vs CDN vs player — so you see which latency dominates.

Across the wire

Cross-machine latency

Measure a genuine SRT contribution between instances, over the real network, with the one-way timing that only synced clocks make honest.

Trade-offs

Codec & config A/B

Sweep any mix of codec, preset, resolution, fps, segment duration, and bitrate; every report embeds its config and diffs directly against another.

Integrity

Timecode survivability

Confirm the QR survives the encode at a given bitrate and resolution before trusting any number — a pass/fail gate, not a guess.

Integrity

Known-delay calibration

Inject a measured delay at one stage and confirm only that stage moves — the correlation check that makes the whole chain believable.

The rig

Hardware & cloud

The validated results run on AWS EC2 — two GPU instances in one VPC — but nothing here is AWS-specific.

Encoder + controller

NVIDIA L40S · g6e

Three NVENC engines drive the split-frame 8K60 encode; the box also hosts the console and the local encode tap.

Mux + packager

NVIDIA L4 · g6

Receives the SRT contribution and packages CMAF / HLS; its NVDEC decodes 8K for the probes — without ever touching the encoder's pace.

AWS glue

VPC + Time Sync

Both instances share one VPC (private-IP, security-group-locked), and the Amazon Time Sync Service keeps chrony µs-accurate — what makes the cross-machine timing honest.

Not AWS-locked

Prism is just Python + FFmpeg + NVENC/NVDEC, so it runs on any Linux GPU host. The AWS pieces map straight across: EC2 GPU instances → GCP A3/G2 or Azure NV/NC-series; the Time Sync Service → any chrony/PTP/NTP source; security groups → the platform's firewall controls. Same rig, different cloud.

Where it stands

Validated so far — and what's left

The core chain is proven end to end at 8K. What's open extends coverage — the last delivery hops and multi-encoder redundancy — and none of it blocks the glass-to-glass number that already works.

Validated

5 confirmed

Demonstrated on real hardware, with pass/fail gates where it counts.

  • Timecode survivability 600 / 600 frames (100%) decoded through 8K HEVC NVENC — the QR carrier survives the encode.
  • Correlation integrity a known +500 ms delay injected at the encode and SRT stages in turn rose alone (T0→T1 +471 ms, T0→T2 +523 ms), with no other stage affected. Injection gate fully closed.
  • Encode holds real-time 8K60 HEVC NVENC, split-frame across 3 engines — sustained 1.0× with zero dropped frames.
  • Distributed chain, correctly ordered T0→T3 across two machines nests as it should — ~3.0 s encode, +0.4 s SRT, +0.1 s packaging, ~3.5 s glass-to-glass. T2 is sample-sparse at 8K but consistent — a stable latency sample.
  • Cross-machine clocks µs-synced to the AWS time source, so the one-way SRT timing is trustworthy.

Still to validate

3 open

Coverage and robustness — none blocking the core number.

  • T4 — CDN stage measure through a real CDN edge; not yet exercised (needs a CDN manifest URL).
  • T5 — player glass-to-glass true source→display latency via the in-browser player decode, end to end.
  • Encoder redundancy & sync confirm a hot-standby encoder stays frame- and timecode-aligned, and exercise failover end to end.

Broadening the instrument

Planned

More formats and modes for the pipelines you actually run.

  • DASH-MPD live measurement — walk DASH manifests directly; endpoint polling is HLS-only today.
  • Audio + A/V-sync — carry an audio track and measure audio latency and lip-sync alongside video.
  • VVC (H.266) support — bring the next-generation codec into the encode ladder.
  • Continuous monitor mode — run as a persistent latency logger with trends, beyond discrete campaigns.

Reach & production

Exploratory

Wider delivery and multi-site scale.

  • Managed-origin delivery — push to S3 / MediaPackage / RTMP(S) targets, not just local CMAF/SRT.
  • Multi-PoP probe fleet — many correlated probes across regions in one view.
Honest by design

Every stage is labeled measured, derived, or not measured — Prism never presents an estimate as a measurement. T1/T2 are measured per-frame on the instrumented path (T2 sample-sparse at 8K); T3–T5 read only when given an origin, CDN, or player URL. What it can't observe cleanly, it says so.
